# Secrets management

Airbyte stores connector credentials and other sensitive configuration in a configured **secret persistence layer**. This page explains how secrets are handled internally.

<Note>
For configuring the secret persistence backend (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault, or Azure Key Vault), see the [deployment secrets configuration](/deploy/secrets) guide.
</Note>

## How Airbyte handles secrets

Connectors declare sensitive fields in their spec using `"airbyte_secret": true`:

```json
{
  "email": { "type": "string" },
  "api_token": { "type": "string", "airbyte_secret": true }
}
```

When a connector is configured, Airbyte:
1. Identifies all fields marked as `airbyte_secret`
2. Generates a deterministic secret coordinate
3. Writes the original secret value to the coordinate in the secret store
4. Replaces the secret field in the stored JSON with a reference to the coordinate

**Secret coordinate format:**
```
airbyte_workspace_<workspace-id>_secret_<uuid>_<version>
```

**Example:** Given the raw configuration:
```json
{
  "email": "itpartners@example.com",
  "api_token": "fake-token"
}
```

Airbyte stores:
```json
{
  "email": "itpartners@example.com",
  "api_token": {
    "_secret": "airbyte_workspace_4e7d7911-0307-40fe-9b79-f00c0dfbb082_secret_d66baab6-3c8d-4ae5-91a6-ca8d904c4780_v1"
  }
}
```

## Secret rules

Airbyte enforces these rules for secret fields:

- Secrets **must not** be returned by the API
- Secrets **must not** be printed in logs
- Secrets **must be obfuscated** in the stored JSON blob and the UI

This allows deployments where operators have infrastructure access without access to connector credentials.

## Secret versioning

When configuration is updated, Airbyte increments the secret coordinate version while preserving the prefix:

```
v1: airbyte_workspace_024eaa40_secret_5675437d_v1
v2: airbyte_workspace_024eaa40_secret_5675437d_v2
```

The stable prefix allows operators to:
- Understand which workspace a secret belongs to
- Track how often a secret has been modified
- Group secrets by connection

## Secret lifecycle

- **On configuration update:** Airbyte deletes old secret versions and creates new ones (as of Airbyte 0.63.10)
- **Temporary secrets:** Secrets created during `DISCOVER` or `CHECK` operations use workspace ID `00000000-0000-0000-0000-000000000000`. Airbyte attempts to expire these after 2 hours.

## Secret persistence backends

Airbyte supports multiple backends for storing secrets:

<CardGroup cols={2}>
  <Card title="AWS Secrets Manager" icon="aws">
    Store secrets in AWS Secrets Manager. Recommended for AWS deployments.
  </Card>
  <Card title="GCP Secret Manager" icon="google">
    Store secrets in Google Cloud Secret Manager. Recommended for GCP deployments.
  </Card>
  <Card title="Azure Key Vault" icon="microsoft">
    Store secrets in Azure Key Vault. Recommended for Azure deployments.
  </Card>
  <Card title="HashiCorp Vault" icon="vault">
    Store secrets in HashiCorp Vault for multi-cloud or on-premises deployments.
  </Card>
</CardGroup>

By default (without a configured backend), Airbyte stores obfuscated secrets in its PostgreSQL database.
